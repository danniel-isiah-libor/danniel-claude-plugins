# Documentation types — outlines

Templates to adapt. Keep the headings readers expect; drop sections that don't apply.

## User guide
1. **Overview** — what it is, who it's for, what you'll be able to do.
2. **Prerequisites** — accounts, permissions, setup.
3. **Getting started** — the first task, end to end.
4. **Tasks** — one section per common task: numbered steps + expected result.
5. **Troubleshooting / FAQ.**
6. **Glossary** (if needed).

## How-to guide (single task)
Goal (one sentence) → Prerequisites → Numbered steps (with screenshots) → Result / verification → Related links.

## Getting-started / tutorial
What you'll achieve → Setup → Guided steps to a first success → "What's next".

## Feature documentation
What the feature does and why it's useful → How to enable/access it → Options & settings (each explained) → Examples → Limits / known issues.

## FAQ
Grouped questions, each a real user question with a short, direct answer; link to fuller docs.

## Release notes
Version + date → Highlights → New features → Improvements → Fixes → Breaking changes / upgrade notes (version derived per `git-conventions` semver). Written for users, not as a commit log.
